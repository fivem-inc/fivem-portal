# fivem-portal 機能ロードマップ

作成日：2026-06-11

---

## 現在実装済みの機能

- 休暇申請・承認フロー（リーダー→マネージャー→経理→社長の多段階）
- 出張報告（GPS記録）
- 交通費申請
- 欠勤・遅刻・早退入力（管理者・承認者のみ）
- 休暇カレンダー（全スタッフ可視）
- アプリ内通知システム（ベルアイコン・バナー）
- ユーザー・グループ管理（管理者のみ）
- レポート・分析タブ
- メール送信機能（Resend / send-email Edge Function / noreply@five-m.com）

---

## 🔴 短期（次の実装候補）

### 1. 残業申請（オーナー希望・最優先）

- **対象**：まずパートタイム従業員向け、将来的に社員も
- **フロー**：申請（日付・時間・理由）→ リーダー承認1段階 → 結果通知
- **DB テーブル案**：`overtime_requests`
  - `user_id`, `request_date`, `hours`, `reason`, `status`（pending/approved/rejected）, `approver_id`, `created_at`
  - 既存の `leave_requests` 構造を流用できるため実装コスト低
- **難易度**：低〜中

---

### 2. 承認フロー通知メール

- 各承認ステップで次の担当者・申請者にメール送信
- 既存の `send-email` Edge Function をそのまま流用
- 通知タイミング案：

```
申請者が申請
  → リーダーにメール「〇〇さんから休暇申請が届きました」
  → リーダー承認
    → マネージャーにメール「〇〇さんの申請がリーダー承認されました」
  → マネージャー受理
    → 申請者にメール「休暇申請が受理されました」
  → 差し戻し時
    → 申請者にメール「差し戻し理由：〇〇」
```

- **難易度**：中

---

### 3. 有給残日数の自動表示

- 取得済み日数から残日数をリアルタイム計算・警告表示
- 年度切り替えにも対応
- **難易度**：低

## 会議審議予定（運用ルール）

- 出張報告：入り報告の要否 / 2名出張時の扱い / 勉強会等への用途拡張
- 休暇申請：申請者へのメール通知追加
- 共通：申請期限・承認者不在時のエスカレーション等

## メモ（エージェント所見）

> 現状は「最小限かつ堅牢」。優れた設計だが、成長フェーズでは給与・シフト管理の統合が急務。
> 残業申請は簡単に追加できるが、その先の「従業員データをいかに活用するか」（給与計算・シフト最適化）を見据えた拡張が重要。
> コード品質は高く、Context API・型定義が整備されているため、新機能追加の敷居は低い。
> 短期は残業申請、中期は給与計算・シフト管理、長期はAIによる最適配置推奨の順で進めることを推奨。
